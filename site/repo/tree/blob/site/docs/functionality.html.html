30 <h1>What the protocol does</h1>
31 <p class="hero-text">
32 Gossip Protocol coordinates useful work through signed transactions, signed execution
33 receipts, peer gossip, and block approval. The implementation separates each resource
34 class into its own balance: HT, CT, ST, and DNS.
35 </p>

79 <h2>Transaction and receipt flow</h2>
80 <ol class="docs-list">
81 <li>A user signs a transaction body with a wallet. The transaction includes the chain ID, nonce, creation time, and transaction kind.</li>
82 <li>A node validates the signature, nonce, balance, and protocol-specific spec limits before accepting it into the mempool.</li>
83 <li>Validators execute work locally or assign leased work to agents. Executors return signed receipts.</li>
84 <li>Nodes gossip transactions, receipts, heartbeat observations, storage proof receipts, blocks, and block approvals to peers.</li>
85 <li>A proposer builds a block from accepted work. Other validators approve exactly one block for a height and view.</li>
86 <li>Finalized state records balance changes, monitor status, storage payments, domain leases, and finalized work outputs.</li>
87 </ol>

112 <section class="docs-section">
113 <h2>Health checks</h2>
114 <p class="copy-block">
115 A health check is a signed request spec for an HTTP target. It can set method, headers,
116 query parameters, timeout, expected status, JSON assertions, header assertions, body
117 assertions, and optional JSON POST body. Validators execute the request and sign
118 <code>SignedHealthReceipt</code> records containing status, latency, assertion results,
119 sampled response headers, sampled body, and errors.
120 </p>
121 </section>
122
123 <section class="docs-section">
124 <h2>Monitors and heartbeats</h2>
125 <p class="copy-block">
126 A monitor turns repeated observations into ledger state. Schedules can be interval-based
127 or UTC cron expressions. Heartbeat monitors accept secret URL pings, delegated-key pings,
128 or dual-auth pings. Misses can be recorded only, confirmed by validators, or confirmed by
129 delegated agents. Browser monitors run a browser journey package on schedule and store
130 browser receipt status per slot.
131 </p>
132 <p class="copy-block">
133 Monitor statuses include up, running, late, down, paused, and insufficient funds. Slot
134 statuses distinguish pending, running, ok, explicit failure, unconfirmed miss, reachable
135 miss, confirmed down, late recovery, paused, and insufficient funds.
136 </p>
137 </section>
138
139 <section class="docs-section">
140 <h2>Browser checks</h2>
141 <p class="copy-block">
142 Browser checks run an approved journey package through a runner program. Supported steps
143 include navigate, click, fill, press, wait for text, assert text, assert URL contains,
144 capture screenshot, open a fresh context, and close a context. A package also defines
145 runtime profile, artifact policy, session cache policy, tags, owner, approver, and
146 approval time.
147 </p>
148 </section>
149
150 <section class="docs-section">
151 <h2>Compute</h2>
152 <p class="copy-block">
153 Compute jobs are sharded. Workloads include integer map operations, Monte Carlo pi, and
154 WASI Preview 1 modules. WASI modules can be sent inline as small hex blobs or referenced
155 from a storage contract through a module reference. Validators reduce matching shard
156 outputs through reducers such as sum, sum squares, count, min/max, Monte Carlo pi, or raw
157 shard outputs.
158 </p>
159 <p class="copy-block">
160 The sandbox policy controls memory, fuel, stdout/stderr sampling, and network access.
161 Production jobs reject network access. Artifact policies let jobs declare required output
162 paths and byte limits for persisted artifacts.
163 </p>
164 </section>
165
166 <section class="docs-section">
167 <h2>Storage and publishing</h2>
168 <p class="copy-block">
169 Storage bundles turn files into chunks with a Merkle root. Encrypted bundles produce a
170 client encryption secret file. Public raw bundles include static-site metadata such as
171 index path, content types, critical prefix hash, and critical prefix byte count. Storage
172 contracts prepay a host for availability, duration, proof cadence, sample count, and
173 reward rate.
174 </p>
175 <p class="copy-block">
176 A storage host serves manifests, chunks, Merkle proofs, and public files. It can also act
177 as a domain gateway by resolving the request host through the validator node's domain-route
178 endpoint and serving the leased storage contract.
179 </p>
180 </section>
181
182 <section class="docs-section">
183 <h2>DNS leases</h2>
184 <p class="copy-block">
185 Validators create domain offerings for suffixes they can serve through a gateway URL.
186 Users lease labels under an offering, bind the lease to a storage contract, renew the
187 lease, rebind it to another contract, or cancel it. A route response resolves host,
188 offering, lease, and target contract metadata.
189 </p>
190 </section>
191
192 <section class="docs-section">
193 <h2>Swaps</h2>
194 <p class="copy-block">
195 Swap adapters quote buy or sell flows against settlement assets. A quote can be locked,
196 cancelled, or settled. The example adapter config uses settlement assets such as USDC and
197 USDT and returns operational follow-up steps for treasury settlement.
198 </p>
