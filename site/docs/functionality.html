What the protocol does

Gossip Protocol coordinates useful work through signed transactions, signed execution receipts, peer gossip, and block approval. The implementation separates each resource class into its own balance: HT, CT, ST, and DNS.

Runtime roles

Role Purpose Main commands
Wallet Creates Ed25519 identities, signs transactions, receipts, leases, and approvals. wallet create, wallet address, wallet show
Validator node Maintains chain state, accepts control transactions, executes local work, gossips receipts, proposes blocks, and approves blocks. node run
Probe agent Executes leased health checks, browser checks, and compute shards for a validator. agent run
Storage host Serves bundled content chunks, Merkle proofs, public raw files, and domain-routed static output. storage host
Client Builds specs and submits signed transactions for transfers, checks, monitors, compute jobs, storage contracts, and DNS leases. tx *, swap *, storage bundle

Transaction and receipt flow

  1. A user signs a transaction body with a wallet. The transaction includes the chain ID, nonce, creation time, and transaction kind.
  2. A node validates the signature, nonce, balance, and protocol-specific spec limits before accepting it into the mempool.
  3. Validators execute work locally or assign leased work to agents. Executors return signed receipts.
  4. Nodes gossip transactions, receipts, heartbeat observations, storage proof receipts, blocks, and block approvals to peers.
  5. A proposer builds a block from accepted work. Other validators approve exactly one block for a height and view.
  6. Finalized state records balance changes, monitor status, storage payments, domain leases, and finalized work outputs.

Token ledgers

HT

Base token used for transfers, health checks, browser checks, monitor budgets, and swap locks.

CT

Compute token used to pay for distributed jobs and reward matching compute receipts.

ST

Storage token used for storage contract prepayment and proof-window rewards.

DNS

DNS token used to pay for subdomain lease duration and public route binding.

Health checks

A health check is a signed request spec for an HTTP target. It can set method, headers, query parameters, timeout, expected status, JSON assertions, header assertions, body assertions, and optional JSON POST body. Validators execute the request and sign SignedHealthReceipt records containing status, latency, assertion results, sampled response headers, sampled body, and errors.

Monitors and heartbeats

A monitor turns repeated observations into ledger state. Schedules can be interval-based or UTC cron expressions. Heartbeat monitors accept secret URL pings, delegated-key pings, or dual-auth pings. Misses can be recorded only, confirmed by validators, or confirmed by delegated agents. Browser monitors run a browser journey package on schedule and store browser receipt status per slot.

Monitor statuses include up, running, late, down, paused, and insufficient funds. Slot statuses distinguish pending, running, ok, explicit failure, unconfirmed miss, reachable miss, confirmed down, late recovery, paused, and insufficient funds.

Browser checks

Browser checks run an approved journey package through a runner program. Supported steps include navigate, click, fill, press, wait for text, assert text, assert URL contains, capture screenshot, open a fresh context, and close a context. A package also defines runtime profile, artifact policy, session cache policy, tags, owner, approver, and approval time.

Compute

Compute jobs are sharded. Workloads include integer map operations, Monte Carlo pi, and WASI Preview 1 modules. WASI modules can be sent inline as small hex blobs or referenced from a storage contract through a module reference. Validators reduce matching shard outputs through reducers such as sum, sum squares, count, min/max, Monte Carlo pi, or raw shard outputs.

The sandbox policy controls memory, fuel, stdout/stderr sampling, and network access. Production jobs reject network access. Artifact policies let jobs declare required output paths and byte limits for persisted artifacts.

Storage and publishing

Storage bundles turn files into chunks with a Merkle root. Encrypted bundles produce a client encryption secret file. Public raw bundles include static-site metadata such as index path, content types, critical prefix hash, and critical prefix byte count. Storage contracts prepay a host for availability, duration, proof cadence, sample count, and reward rate.

A storage host serves manifests, chunks, Merkle proofs, and public files. It can also act as a domain gateway by resolving the request host through the validator node's domain-route endpoint and serving the leased storage contract.

DNS leases

Validators create domain offerings for suffixes they can serve through a gateway URL. Users lease labels under an offering, bind the lease to a storage contract, renew the lease, rebind it to another contract, or cancel it. A route response resolves host, offering, lease, and target contract metadata.

Swaps

Swap adapters quote buy or sell flows against settlement assets. A quote can be locked, cancelled, or settled. The example adapter config uses settlement assets such as USDC and USDT and returns operational follow-up steps for treasury settlement.
